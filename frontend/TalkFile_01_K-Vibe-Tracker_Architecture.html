DB 스키마 설계 (Supabase PostgreSQL)

총 5개 테이블. TourAPI 전체 POI는 저장하지 않고 조회된 것만 캐시하여 500MB 무료 한도 내 운영합니다.

-- 1. 장소 정보 (TourAPI 캐시 + 혼잡도) places ( id, content_id, content_type, name_ko, name_en, name_ja, name_zh, lat, lng, address, category, image_url, crowd_level, -- 0~100 (혼잡도 예측값) crowd_updated_at, created_at, updated_at ) -- 2. SNS 분석 결과 캐시 sns_analysis ( id, url_hash, -- URL MD5 해시 (중복 분석 방지) source_url, -- YouTube / Instagram URL detected_place_id, vibe_tags[], -- ["힙", "빈티지", "감성"] raw_result jsonb, expires_at -- 24시간 TTL ) -- 3. 사용자 페르소나 user_personas ( id, user_id, star_name, mood_tags[], -- ["조용한", "힙"] style_tags[], -- ["모던", "빈티지"] created_at ) -- 4. 생성된 루트 routes ( id, user_id, persona_id, place_ids[], estimated_time, total_distance, share_token, -- 공유 링크용 UUID created_at ) -- 5. 편의시설 (화장실 · 약국 등) facilities ( id, type, -- "restroom" | "cafe_toilet" | "pharmacy" name, address, lat, lng, is_public, is_24h, languages[], -- 지원 언어 표시 source -- "molit" | "tourapi" )
Supabase 무료 한도 주의: 500MB DB 한도 → TourAPI 전체 POI 저장 금지. 사용자가 조회한 것만 upsert 방식으로 캐시. 2주 이상 비활성 프로젝트는 자동 중지됨.

트래픽 관리 및 캐싱 전략

API Rate Limit 보호 흐름

사용자 요청 → Vercel Edge Middleware (IP별 분당 요청 제한: 20 req/min) → Upstash Redis (슬라이딩 윈도우 카운터) → 초과 시 429 Too Many Requests 응답 → 정상 시 Next.js API Route 처리

3-Stage 캐싱 레이어 (비용 절감 핵심)

캐시 레이어구현TTL대상
L1 Vercel Edge Cache Vercel CDN (자동) 30분 정적 응답, 지도 타일, 공통 POI
L2 Upstash Redis Next.js API Route 내 캐싱 24시간 TourAPI 결과, SNS 분석 결과
L3 Supabase DB upsert on miss 영구 (주기 갱신) 장소 정보, 분석 결과, 혼잡도

AI Worker 슬립 방지 전략

// Render.com 무료 플랜: 15분 비활성 시 슬립 // 해결: Vercel Cron으로 10분마다 헬스체크 ping vercel.json: { "crons": [ { "path": "/api/keepalive", "schedule": "*/10 * * * *" // 10분마다 } ] } // /api/keepalive → Render.com AI Worker /health 핑 전송
콜드 스타트 시 30~60초 지연 발생 가능. UI에 로딩 스피너 + "AI 분석 중..." 문구 표시 필수. 슬립 방지 핑으로 실제 사용 중엔 콜드 스타트 최소화.

PERFORMANCE
전처리 최적화 & 위치 정보 오프라인 저장

사전에 계산·저장해두면 실시간 처리 부하를 줄이고 응답 속도를 크게 개선할 수 있는 부분과, GPS 정보를 사용자 기기에 로컬 저장하여 오프라인 또는 저속 네트워크에서도 서비스 품질을 유지하는 방법입니다.

전처리 사전 계산 — 효과가 큰 3가지

01 · TourAPI POI 사전 수집
지역별 인기 장소 DB 사전 적재
문제: 사용자가 지도를 열 때마다 TourAPI 실시간 호출 → 응답 1~2초 지연
해결: GitHub Actions Cron (매일 새벽 3시) 인기 지역 TOP 500 POI 자동 수집 → Supabase 저장
효과: 지도 로딩 시 DB 조회만으로 즉시 응답 (0.1초 이하)
# GitHub Actions schedule
- cron: '0 3 * * *'
# areaBasedList → Supabase upsert
02 · 혼잡도 사전 계산
요일 × 시간대 매트릭스 pre-compute
문제: 런타임에 방문자 통계 집계 → 쿼리 느리고 CPU 사용
해결: 7일(요일) × 24시간 = 168 조합을 장소별로 미리 계산 → crowd_stats 테이블에 저장
효과: 혼잡도 조회 = 단순 SELECT 1건, 응답 10ms 이하
SELECT level FROM crowd_stats
WHERE place_id=? AND dow=?
AND hour=? -- 즉시 반환
03 · AI 도슨트 사전 생성
루트 확정 시 전체 도슨트 미리 생성
문제: 장소 도착 시 OpenAI + TTS 실시간 생성 → 10초+ 대기
해결: 루트 저장 시점에 포함 장소 전체 Background Job으로 미리 생성 → MP3 Supabase Storage 업로드
효과: 도착 → 즉시 MP3 URL 재생 (딜레이 0.5초 이하)
루트 저장 → Background Job
→ 각 place 도슨트 생성
→ Storage에 MP3 캐시

위치 정보 오프라인 — 사용자 기기 저장 방법

Layer 1 · localStorage
마지막 위치 즉시 복원
앱 재시작 시 GPS 재취득 대기(1~3초) 없이 이전 위치를 즉시 사용해 지도를 바로 표시. TTL 30분 초과 시 만료 처리, 백그라운드에서 실제 GPS 업데이트.
// 저장
localStorage.setItem('loc', JSON.stringify(
  {lat, lng, savedAt: Date.now()}))

// 복원 (30분 TTL)
const c = JSON.parse(localStorage.getItem('loc'))
if (c && Date.now()-c.savedAt < 1800000)
  initMap(c.lat, c.lng) // 즉시 표시
Layer 2 · IndexedDB
반경 POI 데이터 오프라인 캐시
현재 위치 반경 2km 내 POI 목록을 IndexedDB에 저장. 지하철·터널 등 오프라인 상황에서도 주변 장소 목록 표시 가능. Service Worker와 연동해 완전한 오프라인 경험 제공.
// idb-keyval 라이브러리 사용
import {set, get} from 'idb-keyval'

// 저장 (위치 그리드 키)
await set(`poi:${gridKey}`, {places, ts})

// 오프라인 시 캐시 사용
const cached = await get(`poi:${gridKey}`)
if (cached) renderPins(cached.places)
